### **2. Experiment A: Precopy Migration (The "Safe" Way)**
**The Logic:** Send RAM while the VM is running on Source. Pause Source, send the last bit, wake up Destination.


**Commands (in Source Monitor `nc 127.0.0.1 4445`):**
1. `migrate_set_parameter max-bandwidth 1M` (Slows it down so you can watch).
2. `migrate -d tcp:127.0.0.1:4444`
3. **Observation:** Source keeps counting. Destination stays black. Once RAM is 99% sent, Source freezes and Destination instantly wakes up.

**Key Code Functions Triggered:**
* `ram_save_iterate` (in `ram.c`): Loops through memory to send pages.
* `migration_completion`: Pauses the Source VM.

---

### **3. Experiment B: Postcopy Migration (The "Magic" Way)**
**The Logic:** Pause Source immediately. Move the CPU state to Destination. Wake up Destination with **zero** RAM. Pull RAM pages from Source only when the CPU tries to read them.


**Commands (Crucial Order):**
1. **Dest Monitor (`nc 127.0.0.1 4446`):** `migrate_set_capability postcopy-ram on`
2. **Source Monitor (`nc 127.0.0.1 4445`):**
   * `migrate_set_capability postcopy-ram on`
   * `migrate_set_parameter max-bandwidth 1M`
   * `migrate -d tcp:127.0.0.1:4444`
   * `migrate_start_postcopy` (Trigger this immediately).

**Observation:** The Source VNC freezes **instantly**. The Destination VNC wakes up **immediately**. The VM is "alive" on the Destination even though the RAM is still being sent over the network.

**Key Code Functions Triggered:**
* `postcopy_ram_incoming_setup` (in `postcopy-ram.c`): Opens the `userfaultfd` handle.
* `postcopy_ram_fault_thread`: The background listener that catches "Missing Page" errors.
* `qemu_ufd_copy_ioctl`: The function that physically "injects" a page into the Destination's RAM.

### **4. Summary Comparison**

| Feature | Precopy (Default) | Postcopy (Your Project Base) |
| :--- | :--- | :--- |
| **When does VM move?** | After almost all RAM is sent. | Before RAM is sent. |
| **Downtime** | Can be high if RAM is "dirtying" fast. | Near-zero (Instant switch). |
| **Risk** | Safe. If network fails, Source stays alive. | Dangerous. If network fails, VM dies. |
| **Primary Tech** | Iterative RAM Looping. | **`userfaultfd` (Page Faults).** |

1. The "Why": Goals & Constraints
The Hardware Constraint: Your system has 4GB RAM. Running two full-sized VMs (like Kali or Ubuntu) simultaneously would trigger an Out-of-Memory (OOM) crash.

The Development Goal: You need to see the internal mechanics of migration (Precopy, Switchover, and State Transfer) using your custom-built QEMU binary (qemu-fast-snapshot).

The Solution: Use an ultra-lightweight Micro-VM (Alpine Linux) and a Headless/VNC setup to keep memory usage under 1GB total.

2. The Architecture ("The What")
The setup uses a Source-Destination model communicating over a local network loopback.

Source VM: The active VM running a "Heartbeat" script.

Destination VM: A "Waiting" instance of QEMU initialized with the -incoming flag.

Control Plane: Two dedicated Telnet monitors for sending commands without log interference.

Data Plane: A TCP socket on port 4444 for memory transfer.

Observation Plane: VNC displays and redirected log files for debugging.
